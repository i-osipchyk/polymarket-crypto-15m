# polymarket-crypto-15m ‚Äî Documentation v1

## Overview

This project is a real-time market data collection and trading system that simultaneously listens to:

- **Binance** WebSocket streams for BTC/USDT (order book depth and trade tape)
- **Polymarket** WebSocket API for the "Bitcoin up or down" 15-minute prediction markets

Raw data from all three streams is aggregated per second and written to disk as combined JSONL files. The project also contains experimental trading bot implementations and Jupyter notebooks for exploratory data analysis.

---

## Goals

1. Collect high-frequency, synchronized microstructure data from both a crypto spot exchange (Binance) and a prediction market (Polymarket).
2. Compute actionable metrics (order book imbalance, aggressor flow imbalance, CVD, etc.) in real time.
3. Persist data in a time-organized folder structure for later analysis and model training.
4. Provide an experimental momentum trading bot that can place orders on Polymarket based on the collected signals.

## Architecture

### Data Flow

```
Binance Depth WS  -->  L2Listener  -->  L2Aggregator  -->  DataManager
Binance Trade WS  -->  TapeListener -> TapeAggregator -->  DataManager  --> JSONLWriter --> data/yyyy/mm/dd/hh/MM_combined_data.jsonl
Polymarket WS     -->  WebSocketOrderBook              -->  DataManager
```

### Threading Model

The entry point (`collector.py`) uses a hybrid concurrency model:

- **asyncio event loop** ‚Äî runs `L2Listener` and `TapeListener` concurrently as coroutines.
- **background thread** ‚Äî runs `polymarket_runner` (the `websocket-client` library is synchronous); it posts data back to the async `DataManager` using `asyncio.run_coroutine_threadsafe`.

### File Rotation

`JSONLWriter` rotates output files every 15 minutes, aligned to wall-clock candle boundaries. Files are stored under `data/yyyy/mm/dd/hh/MM_<name>.jsonl`. The current 15-minute candle start timestamp is computed by `curr_timestamp_15min()`.

## Components

### `L2Listener` ([app/binance/listeners/l2_listener.py](../app/binance/listeners/l2_listener.py))

Connects to `wss://stream.binance.com:9443/ws/btcusdt@depth@100ms`.

- Maintains a full local order book (bids and asks as `{price: qty}` dicts).
- On each update, extracts the top N levels (default: 10) and passes them to `L2Aggregator`.
- Zero-quantity updates remove price levels from the book.
- Reconnects automatically after a 5-second delay on any connection error.
- Malformed messages (missing/wrong fields) are logged and skipped without crashing.
- Incoming messages are capped at 2 MB.

### `TapeListener` ([app/binance/listeners/tape_listener.py](../app/binance/listeners/tape_listener.py))

Connects to `wss://stream.binance.com:9443/ws/btcusdt@trade`.

- Parses each trade: price, quantity, and side (`buy` if maker was seller, `sell` otherwise ‚Äî Binance convention: `m=true` means the buyer was the market maker, i.e. the trade was a sell).
- Passes each trade to `TapeAggregator`.
- Reconnects automatically after a 5-second delay on any connection error.
- Malformed messages are logged and skipped without crashing.
- Incoming messages are capped at 2 MB.

### `L2Aggregator` ([app/binance/aggregators/l2_aggregator.py](../app/binance/aggregators/l2_aggregator.py))

Aggregates L2 snapshots into per-second buckets. When a new second is detected, the previous bucket is finalized and the following metrics are computed:

| Metric | Description |
|---|---|
| `bid_liq` | Total bid-side quantity across top N levels |
| `ask_liq` | Total ask-side quantity across top N levels |
| `obi` | Order Book Imbalance: `(bid - ask) / (bid + ask)` |
| `ew_obi` | Exponentially weighted OBI (alpha=0.2) |
| `bid_liq_delta` | Relative change in bid liquidity vs previous second |
| `ask_liq_delta` | Relative change in ask liquidity vs previous second |
| `weighted_obi` | OBI weighted by inverse distance from mid price |
| `obi_pos_ratio` | Fraction of last 20 seconds with positive OBI |
| `bid_liq_increasing_ratio` | Fraction of last 20 seconds with increasing bid liquidity |
| `ask_liq_decreasing_ratio` | Fraction of last 20 seconds with decreasing ask liquidity |
| `z_<metric>` | Z-score normalized version of selected metrics (rolling window) |

Z-score normalization uses rolling windows (e.g., 300 s for liquidity, 180 s for deltas).

### `TapeAggregator` ([app/binance/aggregators/tape_aggregator.py](../app/binance/aggregators/tape_aggregator.py))

Aggregates trades into per-second buckets. When a new second is detected, finalizes the bucket and computes:

| Metric | Description |
|---|---|
| `price` | Last trade price in the bucket |
| `buy_vol` / `sell_vol` | Aggressively bought / sold volume |
| `total_vol` / `vol_per_sec` | Total traded volume |
| `afi` | Aggressor Flow Imbalance: `(buy - sell) / total` |
| `ew_afi` | Exponentially weighted AFI (alpha=0.2) |
| `cvd` | Cumulative Volume Delta (running sum of buy - sell) |
| `cvd_slope` | Change in exponentially weighted CVD vs previous second |
| `vol_accel` | Volume acceleration: difference of fast (EMA-5) and slow (EMA-20) volume |
| `price_eff` | Price efficiency: price change per unit volume over last 10 s |
| `buy_eff` | Buy efficiency: upward price move per unit buy volume |
| `sell_eff` | Sell efficiency: downward price move per unit sell volume |
| `afi_pos_ratio` | Fraction of last 20 seconds with positive AFI |
| `cvd_slope_pos_ratio` | Fraction of last 20 seconds with positive CVD slope |
| `z_<metric>` | Z-score normalized version of selected metrics (rolling window) |

### `WebSocketOrderBook` / `polymarket_runner` ([app/polymarket/websocket_ob.py](../app/polymarket/websocket_ob.py))

- Fetches the active Polymarket market token IDs for the current 15-minute BTC up/down market from the Gamma API.
- Subscribes to the `market` channel via `wss://ws-subscriptions-clob.polymarket.com/ws/market`.
- Listens for `best_bid_ask` events and maps asset IDs to human-readable outcomes (`up` / `down`).
- Reconnects at each 15-minute candle boundary to pick up the new market's token IDs. Library-level keepalive (`ping_interval=30s`) keeps the connection alive.
- Only whitelisted fields (`outcome`, `best_bid`, `best_ask`) are extracted from server messages ‚Äî the raw message dict is never passed downstream.
- Messages larger than 2 MB are discarded with a warning log.

### `DataManager` ([app/core/data_manager.py](../app/core/data_manager.py))

Merges per-second records from all three sources:

- Accumulates L2, tape, and Polymarket data keyed by Unix timestamp (second).
- When a new timestamp arrives, flushes the completed record to `JSONLWriter`.
- Polymarket data (`up_best_bid`, `up_best_ask`, `down_best_bid`, `down_best_ask`) is updated synchronously from the background thread.
- Outcome values are validated against `{"up", "down"}` before being written ‚Äî unknown outcomes are silently dropped.

### `JSONLWriter` ([app/core/writer.py](../app/core/writer.py))

- Asynchronous queue-based writer.
- Serializes each record as a JSON line.
- Rotates to a new file every 15 minutes aligned to candle boundaries.
- Directory structure: `data/yyyy/mm/dd/hh/MM_<filename>.jsonl`.

## Bots

### `SimpleMomentumLogsBot` ([bots/simple_bot_logs.py](../bots/simple_bot_logs.py))

Simulation / paper-trading bot. Implements a simple momentum strategy:

**Entry conditions** (all must be true):
- BTC price moved up more than `$8.50` in one second.
- Current candle minute is between 5 and 13 (i.e., not too early or too late in the 15-minute window).
- Polymarket "up" ask >= 0.10 and bid < 0.90 (market is not already at an extreme).

**Exit conditions:**
- Stop loss: `entry_ask - 0.02`
- Take profit: `entry_ask + 0.04`

Logs trades only; does not interact with any API.

### `SimpleMomentumAPIBot` ([bots/simple_bot_api.py](../bots/simple_bot_api.py))

Live trading bot. Same strategy as `SimpleMomentumLogsBot`, but:

- Places real buy and take-profit orders via the Polymarket CLOB API (`py_clob_client`).
- Cancels the TP order and places a market-sell if stop loss is triggered.
- Closes open positions when the market token expires at candle rollover.
- Requires `PM_PRIVATE_KEY` and `PM_FUNDER_ADDRESS` in a `.env` file.

## Running the Collector

### Locally

```bash
# Install dependencies
pip install -r requirements.txt

# Set PYTHONPATH to the project root
export PYTHONPATH=.

# Run
python collector.py
```

### With Docker

```bash
docker compose up --build
```

The `data/` and `logs/` directories are volume-mounted so data persists on the host. The `.env` file is automatically loaded by Docker Compose via `env_file`.

## Output Data Format

Each line in `combined_data.jsonl` is a JSON object keyed by metric name. The `timestamp` field is a Unix second. Example fields:

```json
{
  "timestamp": 1740000000,
  "price": 95000.5,
  "buy_vol": 1.23,
  "sell_vol": 0.87,
  "afi": 0.17,
  "ew_afi": 0.12,
  "obi": 0.05,
  "ew_obi": 0.03,
  "weighted_obi": 0.06,
  "cvd": 45.3,
  "up_best_bid": "0.72",
  "up_best_ask": "0.74",
  "down_best_bid": "0.26",
  "down_best_ask": "0.28"
}
```
